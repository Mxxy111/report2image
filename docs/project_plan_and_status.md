# PET/CT 患者友好可视化项目计划与状态

**版本：** 2.0  
**更新日期：** 2026-06-23  
**当前阶段：** 系统与可审计批量流程已具备；研究方案仍处开发迭代，正式 300 例批次尚未完成

## 项目目标

将泌尿系统肿瘤 PET/CT 长报告转换为患者容易理解的医学信息图，准确表达病灶位置、性质倾向、FDG 摄取和 SUVmax，并评估：

1. 无 AI 门控时一次生成的整图通过率；
2. AI 图文审查及反馈修订能否提高同病例通过率；
3. 后续门诊中，图片加文字报告能否改善患者理解度和满意度。

当前癌种为肾癌、前列腺癌和尿路上皮癌。

## 当前研究口径

当前主研究工作草案为 300 例回顾性配对验证：

- 三癌种各 100 例；
- 每例运行 `UNGATED` 和 `GATED`；
- `UNGATED` 只生成一次；
- `GATED` 经过视觉模型审查，并在失败时进行有限次数修订；
- 医生整图 `PASS/FAIL` 是准确性金标准；
- 主要终点为 `UNGATED` 首次图片医生通过率；
- 关键次要终点为同病例两种策略的配对差异，使用精确 McNemar 检验。

旧版“2100 例技术筛查→1500 例保留→三阶段评价”不再作为当前主要研究方案。它可作为项目早期历史或未来独立研究思路，但不得与本轮 300 例结果混合。

统一方案见：

- `docs/research_proposal.md`
- `docs/statistical_analysis_plan.md`
- `docs/image_evaluation_plan.md`
- `docs/decisions/ADR-001-paired-experiment-contract.md`

这些文档当前仍是工作草案。研究团队可以继续调整，但每个实际运行批次应使用独立实验 ID 保留当时版本。

## 已完成能力

### 门诊网页

- 粘贴病例号和 PET/CT 报告；
- 选择 pipeline、provider、模型和生图参数；
- 上传参考样式图；
- 开启或关闭 AI 门控；
- 门控失败后根据具体反馈局部修图；
- 生成患者理解度题目；
- 保存医生整图评价和错误类型；
- 打印患者报告；
- 按日期、病例号和运行 ID 保存结果。
- 查看历史生成记录，并将既往运行重新载入结果页。
- 已提供 Windows 启动器 `NanoBananaPETCT.exe`，可拉起本地网页服务并打开浏览器。
- 门诊场景下，AI 门控定位为“重大错误拦截”而非精细解剖评分；复合区域、双侧发现和多发病灶允许患者友好的合理合并表达，避免因细粒度问题触发耗时重修。
- 默认生图质量调整为 `medium`，优先降低门诊等待时间；如需更高画质，可在本次覆盖参数中手动改为 `high`。

### 可复现元数据

每次运行记录：

- 生图、审查和出题提示词版本及 SHA-256；
- provider、API 类型、Base URL、模型、超时和图像参数；
- 参考图 SHA-256；
- 代码源树 SHA-256 和 Git 状态；
- 实验 ID、工作项 ID、癌种和随机种子；
- `providerSeedApplied`。当前图像 API 不支持 seed，因此该值为 `false`。

### 当前结构化处理策略

生图前的确定性结构化处理主要是 `laterality_plan`，用于把报告中的患者左/右、双侧和中线定位映射到正面人体图的画面侧别。当前后端支持三种模式：

- `script`：只使用程序规则生成左右清单，不调用 AI。用于测试基础左右约束时优先采用，变量最少。
- `auto`：默认模式。先使用程序规则生成左右清单；如果文本/出题模型配置可用，再调用 AI 生成左右清单并与程序结果合并；AI 失败时回退到程序规则。
- `ai`：强制调用文本模型生成左右清单；调用失败或模型输出不可解析时，本次生成失败。

当前网页尚未暴露 laterality 模式选择控件；如需固定模式，需要通过 API 请求传入 `laterality_mode`。在正式比较左右限制效果前，建议先使用 `script` 跑固定病例，确认程序规则本身能避免基本左右错误，再单独比较 `auto` 或 `ai` 是否带来增益。

SUVmax 的处理原则是事实源优先：程序和模型只能使用输入报告原文中已经出现的 SUVmax，不得从 FDG 摄取高低、病灶性质或既往病例估算。已检查当前 CSV 数据，SUV/SUVmax 基本存在于“检查所见”，而“检查结论”中未出现 SUVmax。病例 `2104814` 也是如此：检查结论没有 SUVmax，检查所见中有 `SUVmax=12.9` 和 `SUVmax=2.8`。因此只输入检查结论时，图片不应显示 SUVmax 数值。

后续不再把“检查所见 + 检查结论”当作一段混合文本拆分。项目数据文件本身已经将二者分为两列，未来前端和 API 应改为分别输入 `findings_text` 与 `conclusion_text`：

- `conclusion_text` 是展示范围和临床性质的主来源，决定哪些发现进入图片、优先级、良恶性/不确定倾向和颜色。
- `findings_text` 是证据库，只用于给结论条目安全补充 SUVmax、尺寸、精确位置和原文证据；检查流程、生理性摄取、正常阴性描述和与结论无关的长篇背景不得直接进入生图内容。
- 在 `laterality_plan` 和生图 prompt 之前增加 `display_plan` 步骤：先根据检查结论生成候选展示项，再用每条结论去检查所见中索引最相关句子，合并为去重后的患者可见展示计划。
- `display_plan` 合并可使用 AI 做语义匹配，但必须经过程序校验：SUVmax 必须在被引用的检查所见片段中原样存在；侧别和解剖部位不能冲突；同一解剖区多个 SUVmax 竞争时必须拆成子项或不显示数值；匹配不确定时宁可不补 SUVmax，也不能错配。
- SUVmax 可做轻量视觉化区分，但只表达摄取强弱，不表达良恶性。颜色仍由结论中的性质倾向决定；SUVmax 仅通过病灶点大小、外圈粗细或标签小刻度表达，例如 `<3` 为小点/细外圈、`3-8` 为中点/中等外圈、`>8` 为大点/粗外圈。无论 SUVmax 多高，都不得因此把良性/炎症或不确定发现改成恶性红色。

推荐的 SUVmax 改进路径：

- 短期：保持现有单文本输入，但在操作说明中提示：只输入检查结论时不应期待 SUVmax；需要 SUVmax 必须同时提供检查所见。
- 中期：前端拆成“检查结论”和“检查所见”两个输入框，后端 payload 增加 `conclusion_text` 和 `findings_text`，同时保留 `report_text` 兼容旧接口。
- 长期：实现 `display_plan` 并写入 manifest，生图和审查都优先使用该计划；原始检查所见只作为证据背景，不作为自由绘图材料。

### 统计分析

统计脚本只接受完整真实数据，不再使用模拟数据回退。它强制检查：

- 600 个计划运行全部完成；
- 300 个病例均有两种策略；
- 三癌种各 100 例；
- 全部具有医生 `PASS/FAIL`；
- 模型、参数、提示词、代码哈希和 seed 在批次内一致。

输出总体及癌种通过率、Wilson 95% CI、精确 McNemar 检验、配对率差 bootstrap CI 和错误类型分布。

## 开发迭代与新比较

当前并非最终提交版本，可以继续开展：

- 新生图模型比较；
- 不同审查模型比较；
- 最大修订次数比较；
- 有/无参考图比较；
- 提示词版本比较；

若新增第三种正式策略，需要同时更新：

- `petct/experiment.py` 的策略与工作项定义；
- 批量入口；
- 导出表；
